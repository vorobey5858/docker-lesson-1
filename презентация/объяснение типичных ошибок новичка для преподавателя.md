# Объяснение типичных ошибок новичка для преподавателя

## Как использовать этот файл

Этот документ нужен не как список ошибок, а как сценарий объяснения:

- как быстро распознать проблему;
- что сказать группе;
- как показать решение без перегруза;
- какой вывод закрепить.

## 1. Путают `image` и `container`

Как распознать:

- слушатель говорит, что удалил контейнер и "все пропало";
- пытается запускать контейнер, используя имя старого контейнера как образ.

Что сказать:

"Образ — это заготовка. Контейнер — это уже созданный экземпляр из этой заготовки. Один образ можно использовать много раз."

Как быстро показать:

```bash
docker image ls
docker ps -a
```

Какой вывод закрепить:

- образ и контейнер живут отдельно;
- удаление контейнера не равно удалению образа.

## 2. Думают, что `EXPOSE` сам открывает порт

Как распознать:

- в `Dockerfile` есть `EXPOSE 80`, но слушатель не использует `-p`;
- слушатель удивляется, почему страница не открывается.

Что сказать:

"`EXPOSE` — это описание ожидаемого порта внутри контейнера. Публикация порта наружу делается через `docker run -p`."

Как быстро показать:

```bash
docker run -d --name demo-nginx -p 8080:80 nginx
```

Какой вывод закрепить:

- `EXPOSE` не заменяет `-p`;
- для браузера важен именно порт хоста.

## 3. Забывают точку в `docker build`

Как распознать:

- команда набрана как `docker build -t lesson-site:1.0`;
- Docker ругается на контекст или аргументы.

Что сказать:

"Точка означает текущую папку как контекст сборки. Без контекста Docker не понимает, откуда брать файлы для `COPY`."

Какой вывод закрепить:

- точка — не декоративный символ;
- это важная часть команды.

## 4. Изменили HTML, но не пересобрали образ

Как распознать:

- файл уже изменен;
- в браузере старая страница;
- слушатель уверен, что Docker "показывает кэш".

Что сказать:

"Изменение файла на диске не меняет уже собранный образ. Нужно заново собрать образ и заново запустить контейнер."

Как быстро показать:

```bash
docker build -t lesson-site:1.1 .
docker rm -f lesson-site
docker run -d --name lesson-site-v2 -p 8081:80 lesson-site:1.1
```

Какой вывод закрепить:

- изменил исходники — пересобери образ;
- новый контейнер должен стартовать из нового образа.

## 5. Путают имя контейнера и тег образа

Как распознать:

- вместо образа в `docker run` подставляют имя контейнера;
- смешивают `lesson-site` и `lesson-site:1.0`.

Что сказать:

"Тег относится к образу. Имя относится к контейнеру. Это два разных объекта и два разных уровня."

Какой вывод закрепить:

- образу нужен тег;
- контейнеру нужно имя;
- образ создают, контейнер запускают из образа.

## 6. Пытаются создать контейнер с уже существующим именем

Как распознать:

- Docker пишет, что имя уже используется.

Что сказать:

"Это не страшная ошибка. Docker просто не даст создать второй контейнер с тем же именем. Старый контейнер нужно удалить или назвать новый иначе."

Как быстро показать:

```bash
docker ps -a
docker rm -f lesson-site
```

Какой вывод закрепить:

- имя контейнера должно быть уникальным;
- старые контейнеры часто мешают только потому, что их забыли удалить.

## 7. Открывают не тот порт

Как распознать:

- контейнер работает;
- в `docker ps` все выглядит нормально;
- но в браузере открывают другой порт.

Что сказать:

"Смотрите на левую часть пары `host:container`. Именно этот порт открывают в браузере."

Пример:

- `8081:80` значит, что в браузере нужно открывать `localhost:8081`.

Какой вывод закрепить:

- сначала читаем команду запуска;
- потом проверяем, какой именно порт опубликован на хосте.

## 8. Не читают текст ошибки

Как распознать:

- слушатель сразу говорит "ничего не работает";
- не может пересказать, что именно написал терминал.

Что сказать:

"На первом уровне Docker чаще всего ломается очень буквально. Сначала читаем текст ошибки полностью, потом уже делаем вывод."

Какой вывод закрепить:

- ошибка — это подсказка;
- сначала читаем сообщение, потом проверяем путь, имя, порт и команду.

## 9. Не смотрят `docker logs`

Как распознать:

- браузер не открывается или работает странно;
- слушатель не открывал логи контейнера.

Что сказать:

"Браузер показывает только внешний эффект. `docker logs` показывает, что контейнер сообщает о своей работе."

Как быстро показать:

```bash
docker logs demo-nginx
```

Какой вывод закрепить:

- логи — это нормальный инструмент диагностики;
- привычка смотреть логи нужна с первого урока.

## 10. Пугаются маленькой ошибки

Как распознать:

- слушатель быстро перестает пробовать;
- после первой ошибки говорит, что тема слишком сложная.

Что сказать:

"Ошибка на таком уроке — это не провал, а часть практики. Наоборот, если вы сейчас поймете и исправите маленькую ошибку, дальше будет гораздо спокойнее."

Какой вывод закрепить:

- большинство первых проблем в Docker простые;
- уверенность приходит не когда ошибок нет, а когда человек умеет их читать и чинить.

## Общая педагогическая стратегия

Когда у студента ошибка:

1. Сначала попросите его прочитать сообщение вслух.
2. Затем уточните: путь, порт, имя контейнера, тег образа, контекст сборки.
3. Только после этого подсказывайте решение.

Лучший стиль реакции преподавателя:

- спокойно;
- без оценки;
- с акцентом на логику поиска, а не на сам факт ошибки.
