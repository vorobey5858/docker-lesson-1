# План урока: Docker и контейнеризация #1

## 1. Общая идея урока

**Тема:** Docker и контейнеризация приложений  
**Уровень:** начинающие ИТ-специалисты  
**Формат:** теория + живая демонстрация + практика + мини-кейс + разбор ошибок  
**Рекомендуемая длительность:** 180 минут  
**Финальный результат:** к концу занятия слушатели понимают, зачем нужна контейнеризация, различают `image` и `container`, ориентируются в базовой архитектуре Docker, знают роль `namespaces` и `cgroups`, умеют собрать и запустить свой первый образ через `Dockerfile`, а также могут найти и исправить простую типичную ошибку.

## 2. Цель урока

Показать начинающим специалистам, зачем нужна контейнеризация, чем контейнер отличается от виртуальной машины, как устроен Docker на базовом уровне, какие механизмы изоляции лежат в основе контейнеров и как собрать простой образ на базе `nginx`, а затем запустить его как веб-контейнер.

## 3. Учебные результаты

После урока слушатель должен:

- объяснять простыми словами, что такое контейнеризация приложений;
- понимать, какую проблему решают контейнеры в разработке и эксплуатации;
- различать понятия `image`, `container`, `registry`, `Dockerfile`, `Docker Engine`, `container runtime`;
- понимать базовую цепочку `CLI -> daemon -> image/runtime -> container`;
- понимать на интуитивном уровне, что делают `namespaces` и `cgroups`;
- уметь запускать готовый контейнер, смотреть список контейнеров, логи и базово управлять жизненным циклом;
- уметь читать и писать простой `Dockerfile`;
- уметь собрать собственный образ и запустить контейнер из него;
- уметь распознать несколько типичных ошибок новичка и исправить их.

## 4. Входные требования

Перед уроком желательно, чтобы у слушателей уже были:

- базовое понимание, что такое процесс, приложение, ОС и команда в терминале;
- минимальный опыт работы с командной строкой;
- общее представление о веб-приложении, HTML-странице или статическом сайте.

Если аудитория совсем новая, в начале занятия нужно отдельно пояснить:

- что такое образ;
- что такое процесс;
- что такое порт;
- что значит "запустить приложение в изолированной среде";
- почему команда в терминале может завершиться сразу или работать долго.

## 5. Что подготовить до урока

### 5.1. Техническая подготовка

Нужно заранее подготовить среду, чтобы практика не сорвалась:

- установить `Docker Desktop` или `Docker Engine` на компьютер преподавателя;
- проверить, что команды `docker version` и `docker run hello-world` работают;
- заранее скачать образы `hello-world` и `nginx`, если есть риск нестабильного интернета;
- проверить, что порт `8080` свободен;
- открыть редактор кода и терминал заранее;
- подготовить рабочую папку с практическими файлами;
- проверить, что браузер открывает `http://localhost:8080`;
- иметь резервный вариант на случай, если Docker не стартует у части группы.

### 5.2. Учебные материалы

К уроку должны быть готовы:

- презентация PowerPoint;
- расширенная шпаргалка преподавателя;
- готовая папка `docker-lesson-1` для основной практики;
- папка `docker-lesson-1-mini-case` для самостоятельного мини-кейса;
- `README.md` с командами и шагами;
- отдельный список типичных ошибок новичка;
- чеклист преподавателя перед уроком;
- набор контрольных вопросов для середины и конца занятия.

### 5.3. Практические файлы

Для основной практики удобно использовать такую структуру:



## 7. Теоретическое содержание урока

### 7.1. Почему появилась контейнеризация

Ключевые тезисы:

- одна и та же программа может вести себя по-разному на разных машинах;
- проблема часто не в коде, а в окружении;
- версии интерпретаторов, библиотек, системных пакетов и переменных окружения влияют на запуск;
- контейнеризация повышает воспроизводимость и предсказуемость запуска;
- контейнер не делает приложение автоматически "правильным", но делает его запуск более управляемым.

### 7.2. Контейнеры и виртуальные машины

Нужно донести:

- контейнер и VM решают похожую задачу, но разным способом;
- контейнеры обычно легче и быстрее стартуют;
- VM обычно тяжелее, но дают более жесткую изоляцию;
- Linux-контейнеры используют механизмы Linux;
- на Windows и macOS при работе через `Docker Desktop` Linux-контейнеры идут внутри Linux VM.

### 7.3. Базовые сущности Docker

Нужно закрепить:

- `image` — шаблон или заготовка;
- `container` — запущенный экземпляр образа;
- `registry` — удаленное хранилище образов;
- `Dockerfile` — текстовое описание сборки образа.

Удобная формула для аудитории:

`image -> шаблон, container -> экземпляр, Dockerfile -> рецепт сборки`

### 7.4. Архитектура Docker

Опорные понятия:

- `Docker CLI` — точка входа пользователя;
- Docker daemon — служба, которая реально обрабатывает команды;
- `Docker Engine` — связка CLI, API и `dockerd`;
- `container runtime` — компонент запуска контейнера;
- registry — место хранения и доставки образов;
- локальное хранилище — уже скачанные образы на машине.

Базовая цепочка:

`пользователь -> Docker CLI -> daemon -> image/runtime -> container`

### 7.5. Container Engine и Container Runtime

Короткая формула:

- engine управляет;
- runtime запускает.

Важно сказать вслух:

- Docker использует внутренние компоненты вроде `containerd` и `runc`;
- новичку не нужно сейчас знать OCI-детали, но полезно понимать разделение ролей.

### 7.6. Механизмы изоляции: `namespaces`

Главная идея:

- контейнеру как будто показывают отдельную картину мира;
- процессы, сеть, hostname, файловая система и пользователи могут быть изолированы;
- это не магия, а набор механизмов Linux.

Нужно кратко упомянуть:

- `PID`
- `NET`
- `MNT`
- `IPC`
- `UTS`
- `USER`

### 7.7. Механизмы изоляции: `cgroups`

Главная идея:

- контейнер нужно не только изолировать, но и ограничивать по ресурсам;
- можно ограничивать CPU, память, количество процессов;
- это защищает хост и другие контейнеры от "жадного" приложения.

Подходящий пример для вслух:

`docker run --cpus=1 --memory=256m nginx`

### 7.8. Dockerfile

Ключевая мысль:

- `Dockerfile` описывает сборку образа, а не работающий контейнер;
- каждая инструкция отвечает на отдельный вопрос сборки;
- контекст сборки — это файлы, которые команда `docker build` видит в текущей директории.

Нужно проговорить:

- `FROM` — от чего строим;
- `COPY` — что переносим в образ;
- `WORKDIR` — где работаем;
- `RUN` — что выполнить на этапе сборки;
- `EXPOSE` — какой порт документируем;
- `CMD` — что запускается по умолчанию.

Анти-миф:

`EXPOSE` сам по себе не публикует порт наружу.

## 8. Практическая часть

### 8.1. Демо 1: первый запуск контейнера

Цель:

- показать, что образ может скачаться автоматически;
- показать, что контейнер может завершиться сразу;
- зафиксировать разницу между `docker ps` и `docker ps -a`.

Команды:

```bash
docker run hello-world
docker ps
docker ps -a
docker images
```

Вопросы группе перед запуском:

- Что, по-вашему, произойдет после `docker run hello-world`?
- Увидим ли мы контейнер в `docker ps`?
- Чем будет отличаться `docker ps -a`?

### 8.2. Демо 2: запуск веб-контейнера

Цель:

- показать долго живущий контейнер;
- показать проброс порта;
- показать логи и жизненный цикл.

Команды:

```bash
docker run -d --name demo-nginx -p 8080:80 nginx
docker ps
docker logs demo-nginx
docker stop demo-nginx
docker rm demo-nginx
```

Что должны увидеть слушатели:

- контейнер работает в фоне;
- страница открывается в браузере;
- логи показывают обращения к `nginx`;
- контейнер можно остановить и удалить отдельно от образа.

### 8.3. Основная практика: собираем свой образ

Цель:

- руками создать простой контейнеризированный сайт;
- закрепить структуру проекта;
- почувствовать цикл "изменил -> пересобрал -> перезапустил -> проверил".

Шаги:

1. Открыть папку `docker-lesson-1`.
2. Посмотреть `index.html` и `Dockerfile`.
3. Выполнить сборку образа.
4. Запустить контейнер.
5. Открыть страницу в браузере.
6. Изменить заголовок страницы.
7. Пересобрать образ.
8. Перезапустить контейнер.
9. Проверить обновленный результат.

Команды:

```bash
docker build -t lesson-site:1.0 .
docker run -d --name lesson-site -p 8080:80 lesson-site:1.0
docker ps
docker stop lesson-site
docker rm lesson-site
docker build -t lesson-site:1.1 .
docker run -d --name lesson-site-v2 -p 8081:80 lesson-site:1.1
```

Что важно проговорить:

- точка в конце `docker build` означает текущий контекст сборки;
- после изменения файлов старый образ сам не обновляется;
- контейнер создается из уже собранного образа;
- имя контейнера и тег образа — это разные сущности.

### 8.4. Мини-кейс на самостоятельное исправление ошибки

Цель:

- вывести слушателей из режима "повторить за преподавателем";
- научить не пугаться маленьких ошибок;
- дать первый опыт поиска проблемы.

Подход:

- дать папку `docker-lesson-1-mini-case`;
- попросить собрать проект;
- не подсказывать сразу, что именно сломано;
- дать 3-5 минут на попытку найти причину;
- затем сделать общий разбор.

Подходящие ошибки для первого урока:

- неверный путь в `COPY`;
- забытая точка в `docker build`;
- неправильный порт;
- уже существующий контейнер с тем же именем;
- изменили файл, но открывают старый контейнер.

## 13. Как сделать урок действительно интересным

Чтобы 3 часа не превратились в усталую лекцию, преподавателю важно:

- просить группу предсказывать результат команд до запуска;
- давать короткие вопросы после каждого крупного блока;
- один раз специально показать маленькую ошибку и исправить ее вместе;
- позволить участникам персонализировать страницу в практике;
- завершать каждый смысловой блок одной короткой формулой.

Подходящие формулы для урока:

- контейнеризация делает запуск воспроизводимее;
- `image` — шаблон, `container` — экземпляр;
- engine управляет, runtime запускает;
- `namespaces` отвечают за изоляцию, `cgroups` — за ресурсы;
- `Dockerfile` описывает сборку образа;
- изменил файлы — пересобери образ.
